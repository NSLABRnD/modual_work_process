### 1.2 핵심 문제점 (Pain Points)

| # | 문제 | 설명 |
|---|------|------|
| P1 | **특정 인원 의존성** | 특정 팀원만 처리 가능한 업무가 많아 병목(Bottleneck) 발생. 담당자 부재 시 업무 마비 |
| P2 | **코드 재사용 어려움** | 프로젝트/블록체인마다 유사 코드를 매번 새로 작성. 중복 작업 발생 |
| P3 | **협업/인터페이스 부재** | 팀원 간 작업 결과물의 연동 기준이 없어 통합 시 충돌 및 지연 |
| P4 | **지식 사일로(Silo)** | 블록체인별, 플랫폼별 지식이 개인에게 고립되어 있음 |

## 2. 목표 정의 (To-Be)

### 2.1 상위 목표

MOSA(Modular Open Systems Approach) 원칙을 블록체인 소프트웨어 개발팀의 **업무 프로세스**와 **소프트웨어 아키텍처** 양 측면에 적용하여:

1. **모듈화** - 업무와 코드를 독립적인 모듈 단위로 분리
2. **표준화** - 모듈 간 인터페이스와 협업 규칙을 표준화
3. **재사용성** - 모듈의 재활용으로 중복 작업 제거
4. **교체 가능성** - 특정 인원/기술에 종속되지 않는 구조

### 2.2 MOSA 5대 핵심 원칙 매핑

| MOSA 원칙 | 우리 팀 적용 방향 |
|----------|--------------|
| **원칙 1: 프레임워크 지원 환경** | 팀 개발 가이드, 온보딩 문서, 교육 체계 수립 |
| **원칙 2: 모듈식 설계 적용** | 소프트웨어를 기능 단위 모듈로 분리 (높은 응집도, 낮은 결합도) |
| **원칙 3: Key Interface 제공** | 모듈 간 표준 인터페이스(API) 정의 |
| **원칙 4: 개방형 표준 적용** | 오픈소스/업계 표준 기술 우선 채택 |
| **원칙 5: 적합성 인증** | 모듈별 테스트/검증 체계 구축 |

## 3. 요구사항 상세

### 3.1 업무 프로세스 모듈화 요구사항

#### R-P1: 업무 도메인 분류 체계

업무를 독립적 모듈 단위로 분류하여 팀원이 교체 가능하도록 설계한다.

| 모듈 ID | 도메인 | 주요 업무 | 필요 스킬 |
|---------|--------|---------|---------|
| WM-01 | **블록체인 인터랙션** | 체인별 RPC/SDK 연동, 트랜잭션 처리, 이벤트 리스닝 | Web3, ethers.js 등 |
| WM-02 | **스마트컨트랙트** | 컨트랙트 설계/개발/배포/관리 | Solidity, Rust, Move 등 |
| WM-03 | **백엔드 서버** | API 서버, DB 관리, 가격정보/알람/광고 서비스 | Node.js, PostgreSQL |
| WM-04 | **모바일 앱** | 크로스플랫폼 앱 개발 (Flutter 중심, 네이티브 보조) | Flutter/Dart, Swift, Kotlin |
| WM-05 | **인프라/DevOps** | 서버 배포, 모니터링, CI/CD, 블록체인 노드 관리 | Docker, SSH, Linux |
| WM-06 | **보안/암호화** | 키 관리, 서명, 암호화 로직, 감사(Audit) | 암호학, HSM |

#### R-P2: 역할 매트릭스 (RACI 기반)

- 각 모듈별 **주담당(Primary)** 1명 + **부담당(Secondary)** 1명 이상 배정
- 어떤 업무도 1명만 수행 가능한 상태(Single Point of Failure)가 되지 않도록 설계
- 분기별 역할 로테이션을 통해 지식 확산

#### R-P3: 업무 산출물 표준화

- 모든 모듈의 산출물은 정의된 **인터페이스 문서 양식**을 따름
- 코드 리뷰 시 인터페이스 준수 여부를 체크리스트에 포함
- 각 모듈은 독립적으로 개발/테스트/배포가 가능해야 함

#### R-P4: 지식 공유 체계

- 모듈별 기술 문서(Wiki/Docs) 필수 작성
- 주간/격주 기술 공유 세션 운영
- 온보딩 문서를 통해 신규 인원이 2주 내 모듈 투입 가능하도록

### 3.2 소프트웨어 아키텍처 모듈화 요구사항

#### R-A1: 계층형 아키텍처 (Layered Architecture)

MOSA의 모듈식 설계 원칙에 따라 소프트웨어를 다음 계층으로 분리한다:

```
┌─────────────────────────────────────────────────────┐
│                 Application Layer                    │
│   (각 프로젝트별 비즈니스 로직, UI)                      │
├─────────────────────────────────────────────────────┤
│                  Service Layer                       │
│   (도메인 서비스: 가격조회, 알람, 광고 등)                │
├─────────────────────────────────────────────────────┤
│               Core Module Layer                      │
│   (공통 모듈: 지갑, 트랜잭션, 서명, 인증 등)              │
├─────────────────────────────────────────────────────┤
│              Blockchain Adapter Layer                 │
│   (체인별 어댑터: Ethereum, Solana, Klaytn 등)         │
├─────────────────────────────────────────────────────┤
│               Infrastructure Layer                   │
│   (DB, 캐시, 메시지큐, 외부 API 연동)                   │
└─────────────────────────────────────────────────────┘
```

#### R-A2: 블록체인 어댑터 패턴 (Key Interface)

여러 블록체인과의 상호작용을 **표준 인터페이스**로 추상화한다.

```
[Key Interface: IBlockchainAdapter]
├── connect(config) → Connection
├── getBalance(address) → Balance
├── sendTransaction(txData) → TxResult
├── deployContract(bytecode, abi) → ContractAddress
├── callContract(address, method, params) → Result
├── listenEvents(address, eventName) → EventStream
└── getTransactionStatus(txHash) → Status
```

- 각 블록체인(Ethereum, Solana, Klaytn, Polygon 등)은 이 인터페이스를 구현
- 신규 블록체인 추가 시 어댑터만 구현하면 상위 로직 수정 불필요

#### R-A3: 스마트컨트랙트 모듈화

| 구분 | 설명 |
|------|------|
| **공통 컨트랙트 템플릿** | 토큰(ERC-20/721), 거버넌스, 멀티시그 등 재사용 가능한 템플릿 |
| **체인별 배포 파이프라인** | 하나의 컨트랙트 로직 → 체인별 컴파일/배포 자동화 |
| **ABI 레지스트리** | 배포된 컨트랙트의 ABI/주소를 중앙 관리 |

#### R-A4: 백엔드 서비스 모듈화

기존 모놀리식 서버를 기능 단위 모듈로 분리:

| 모듈 | 기능 | 인터페이스 |
|------|------|----------|
| **price-service** | 암호화폐 가격 정보 수집/제공 | REST API / WebSocket |
| **notification-service** | 알람/푸시 발송 | REST API / Event Queue |
| **ad-service** | 광고 관리 및 노출 | REST API |
| **auth-service** | 인증/인가 | REST API / JWT |
| **blockchain-gateway** | 블록체인 어댑터 래핑 | REST API / gRPC |

#### R-A5: 모바일 앱 모듈화

- **Flutter 패키지 구조**: feature별 독립 패키지로 분리
- **공통 UI 컴포넌트 라이브러리**: 디자인 시스템을 패키지화
- **플랫폼 채널 추상화**: 네이티브(iOS/Android) 기능을 표준 인터페이스로 래핑

#### R-A6: 공통 라이브러리 (Core Modules)

프로젝트 간 재사용을 위한 공통 모듈 목록:

| 모듈 | 용도 | 대상 |
|------|------|------|
| `@core/crypto` | 암호화/복호화, 키 관리, 서명 | 전 프로젝트 |
| `@core/wallet` | 지갑 생성/관리, 니모닉 | 전 프로젝트 |
| `@core/chain-adapter` | 블록체인 어댑터 인터페이스 | 백엔드/프론트엔드 |
| `@core/types` | 공통 타입/인터페이스 정의 | 전 프로젝트 |
| `@core/utils` | 유틸리티 함수 (주소 변환, 단위 변환 등) | 전 프로젝트 |

### 3.3 인프라/도구 요구사항

#### R-I1: 버전 관리 표준

- **Monorepo** 또는 **Multi-repo + 패키지 레지스트리** 중 선택 필요
- Git branching 전략 표준화 (Git Flow or Trunk-based)
- 커밋 메시지 컨벤션 통일

#### R-I2: CI/CD 파이프라인

- 모듈별 독립 빌드/테스트/배포
- 스마트컨트랙트: 테스트넷 자동 배포 + 검증
- 백엔드: 스테이징 → 프로덕션 단계별 배포
- 모바일: 자동 빌드 + 테스트 플라이트/내부 배포

#### R-I3: 문서화 시스템

- API 문서 자동 생성 (OpenAPI/Swagger)
- 컨트랙트 문서 자동 생성 (NatSpec)
- 내부 위키 (기술 결정, 아키텍처 문서)

#### R-I4: 모니터링/관찰가능성

- 서비스별 로깅/메트릭 표준
- 블록체인 트랜잭션 모니터링
- 장애 알림 체계

## 4. 모듈화 디자인 원칙 (MOSA 기반)

### 4.1 적절한 규모 (Right-Sizing)
- 각 모듈은 **self-contained**: 독립적으로 개발, 테스트, 배포 가능
- 너무 크지도, 너무 작지도 않은 적절한 규모

### 4.2 높은 응집도 (High Cohesion)
- 하나의 모듈은 하나의 책임만 수행
- 변경 시 영향받는 모듈 수를 최소화

### 4.3 낮은 결합도 (Low Coupling)
- 모듈 간 의존성은 **인터페이스(Key Interface)**를 통해서만
- 구현체가 아닌 추상화에 의존

### 4.4 개방형 표준 우선
- 라이선스 비용이 없는 오픈 표준 기술 우선 채택
- 특정 벤더/플랫폼 종속 최소화

### 4.5 적합성 검증
- 모듈별 단위 테스트 필수
- 인터페이스 준수 여부 자동 검증 (Contract Testing)
- 코드 리뷰 시 모듈화 원칙 준수 체크리스트 활용

## 7. 참고: MOSA 원칙 요약 (6.LIG.pdf 기반)

> **MOSA 정의**: 모듈식 설계를 채택하고, 주요 인터페이스를 정의하는 통합 비즈니스 및 기술 전략.
> 필요한 경우, 공인된 산업 표준 기구에서 게시하고 유지 관리하는 널리 지원되는 합의 기반 표준을 사용.

**핵심 가치**:
- 모듈화와 개방형 표준 적용으로 **최소한의 노력**으로 기존 시스템을 조합/개량
- 특정 플랫폼/기술/업체에 **종속되지 않는** 아키텍처
- **신속한 기능 확장**과 **비용 절감**을 동시에 달성

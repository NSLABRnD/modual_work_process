I. 현황 분석

현재 업무 방식의 핵심 문제점

⚠

특정 인원 의존성 (SPOF)

특정 팀원만 처리 가능한 업무가 많아 병목 발생.
담당자 부재 시 전체 업무가 멈추는 위험 상존.

🔄

코드 재사용 불가

프로젝트마다 유사한 코드를 매번 새로 작성.
블록체인별 중복 작업이 반복적으로 발생.

🔌

협업 인터페이스 부재

팀원 간 작업 결과물의 연동 기준이 없어
통합 시 충돌과 지연이 빈번하게 발생.

🔒

지식 사일로 (Knowledge Silo)

블록체인별, 플랫폼별 지식이 개인에게 고립.
팀 전체 역량으로 축적되지 않음.

I. 현황 분석

As-Is vs To-Be 비교

● 현재 (As-Is)

  • 업무가 개인에게 종속된 구조
  • 프로젝트마다 처음부터 개발
  • 연동 규칙 없이 각자 방식대로
  • 문서화 부재, 구두 인수인계
  • 한 명이 빠지면 일정 지연
  • 블록체인마다 별도 코드베이스
➜

● 목표 (To-Be)

  • 업무가 모듈 단위로 분리, 교체 가능
  • 공통 모듈을 조합하여 빠르게 개발
  • 표준 인터페이스로 자연스러운 연동
  • 모듈별 기술 문서 필수 작성
  • 주/부 담당 체계로 리스크 분산
  • 어댑터 패턴으로 체인 추상화

III. 프로세스 모듈화

팀 업무를 6개 독립 모듈로 분류합니다. 각 모듈은 주담당 + 부담당 체계로 운영됩니다.

모듈 ID 도메인 주요 업무 기술 스택 담당 구조
WM-01 블록체인 인터랙션 체인별 RPC/SDK 연동, TX 처리, 이벤트 리스닝 Web3, ethers.js 등 주 1 + 부 1
WM-02 스마트컨트랙트 컨트랙트 설계/개발/배포/관리 Solidity, Rust, Move 주 1 + 부 1

WM-05 인프라/DevOps 서버, CI/CD, 블록체인 노드 관리 Docker, SSH, Linux 주 1 + 부 1
WM-06 보안/암호화 키 관리, 서명, 암호화 로직, Audit 암호학, HSM 주 1 + 부 1

IV. 아키텍처 모듈화

계층형 아키텍처 설계

소프트웨어를 5개 계층으로 분리하여 모듈의 재활용성과 이식성을 극대화합니다.

Application Layer 각 프로젝트별 비즈니스 로직, UI — 프로젝트마다 고유한 부분
Service Layer 도메인 서비스 (가격조회, 알람, 광고, 인증) — 재사용 가능한 서비스
Core Module Layer 공통 핵심 모듈 (지갑, 트랜잭션, 서명, 암호화) — 전 프로젝트 공유
Blockchain Adapter Layer 체인별 어댑터 (Ethereum, Solana, Klaytn...) — Key Interface로 추상화
Infrastructure Layer DB, 캐시, 메시지큐, 외부 API — 인프라 기반

설계 원칙: 상위 계층은 하위 계층에만 의존하며, 같은 계층 내 모듈은 인터페이스를 통해서만 통신합니다. 하위 계층일수록 변경 빈도가 낮고 안정성이 높습니다.

IV. 아키텍처 모듈화

Key Interface: 블록체인 어댑터 패턴

여러 블록체인과의 상호작용을 하나의 표준 인터페이스로 추상화합니다.

interface IBlockchainAdapter {
  connect(config): Connection
  getBalance(address): Balance
  sendTransaction(txData): TxResult
  deployContract(bytecode, abi): Address
  callContract(addr, method, params): Result
  listenEvents(addr, event): EventStream
  getTxStatus(txHash): Status
}

어댑터 구현 체인

EthereumAdapter ethers.js / web3.js
SolanaAdapter @solana/web3.js
KlaytnAdapter caver-js
PolygonAdapter ethers.js (EVM)
NewChainAdapter 신규 체인 추가 시

신규 블록체인 추가 시 어댑터만 구현하면 상위 로직 수정 불필요

Q1. 모듈화의 대상 범위

모듈화가 적용되어야 하는 대상은 어디까지인가?

A. 업무 프로세스의 모듈화

팀원의 업무 할당/진행 방식을 모듈 단위로 표준화

B. 소프트웨어(결과물)의 모듈화

개발 결과물인 코드/시스템 구조 자체를 모듈화

현재 제안: 양쪽 모두 적용 (프로세스 모듈화 → 아키텍처 모듈화 순으로 단계적 도입)

Q2. 표준화의 깊이 (Depth of Standardization)

SW 모듈화 시, 표준화를 어느 수준까지 적용할 것인가?

A
국제 표준 수준

외부 개발자 협업, Exit(소스코드 매각) 시에도 호환 가능한 국제 표준 기반 설계

B
사내 표준 수준 (NS Lab 내부)

사내 타 프로젝트 간 연동 용이 (예: PureWallet ↔ PureCertificate ↔ PurePet)

C
프로젝트 단위 표준 (PureWallet 중심)

현재 집중 개발 중인 PureWallet 프로젝트 내부 모듈화 표준

표준화 범위 포함 관계:  A ⊃ B ⊃ C  — A가 가장 넓은 범위이며, B는 A에 포함, C는 B에 포함됩니다. 범위가 넓을수록 초기 투자 비용이 크지만 장기적 확장성이 높습니다.

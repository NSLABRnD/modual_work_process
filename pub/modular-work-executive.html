핵심 개념

모듈화란 무엇인가?

자동차 공장을 생각해보겠습니다.

현재 방식은 차를 만들 때마다 엔진, 바퀴, 시트를 매번 처음부터 직접 깎아 만드는 것과 같습니다.

모듈화 방식은 엔진, 바퀴, 시트를 규격화된 부품으로 만들어 놓고, 필요할 때 조합해서 다양한 차량을 빠르게 만드는 것입니다.

부품(모듈)은 독립적으로 개선할 수 있고, 다른 차종에도 재사용할 수 있습니다.

🧩
모듈 = 독립된 부품
혼자서도 동작하는
완결된 기능 단위
🔗
인터페이스 = 연결 규격
모듈끼리 연결하는
표준화된 규칙
🚀
조합 = 빠른 개발
기존 부품을 조립해
새 제품을 신속하게 출시
적용 가능 범위

모듈화가 적용될 수 있는 두 가지 차원

👥

1. 업무 방식의 모듈화

팀원들의 일하는 방식을 체계화하는 방향입니다.

✓  업무를 6개 전문 영역으로 분류
✓  각 영역에 주담당 + 부담당 배치
✓  분기별 역할 교대로 지식 확산
✓  표준 프로세스로 일 진행
💻

2. 소프트웨어의 모듈화

개발 결과물의 구조를 체계화하는 방향입니다.

✓  공통 기능을 재사용 부품으로 추출
✓  블록체인별 코드를 통일된 규격으로
✓  서비스를 독립 단위로 분리
✓  새 프로젝트는 부품 조합으로 시작

두 가지를 동시에 또는 한쪽만 적용할 수도 있으며, 적용 범위와 깊이에 대한 판단이 필요합니다.

변화 비교

현재 vs 모듈화 적용 시 예상 변화

현재 (As-Is)

  • 업무가 개인에게 종속
  • 프로젝트마다 처음부터 개발
  • 합치는 기준이 없어 충돌
  • 문서 없이 구두 인수인계
  • 1명이 빠지면 일정 지연
  • PureWallet, PurePet 등
    프로젝트 간 코드 공유 불가
➜

모듈화 적용 시 (To-Be)

  • 업무가 모듈 단위, 누구나 투입 가능
  • 공통 부품을 조합해 빠르게 개발
  • 표준 규격으로 자연스럽게 연결
  • 모듈별 문서가 필수로 작성됨
  • 주/부 담당 체계로 리스크 분산
  • 프로젝트 간 핵심 모듈 공유

참고 사례

MOSA: 미국방부가 채택한 모듈화 전략

이번 검토는 미국방부의 MOSA(Modular Open Systems Approach) 원칙을 참고하여 분석했습니다.

MOSA는 미국방부가 2020년 법적으로 의무화한 시스템 개발 방법론입니다.

핵심은 “시스템을 독립된 모듈로 만들고, 표준 인터페이스로 연결하여,
특정 업체/기술에 종속되지 않으면서 빠르게 확장할 수 있도록” 하는 것입니다.

우리 팀의 상황 — 여러 블록체인, 여러 플랫폼, 다양한 언어 — 에 정확히 부합하는 접근법입니다.

2020

Q1. 모듈화의 대상

모듈화가 적용되어야 하는 대상은 어디까지입니까?

A. 업무 프로세스

B. 소프트웨어 결과물

코드/시스템 구조 자체를 모듈화

A만, B만, 또는 A+B 모두 적용 가능합니다. 어떤 범위까지 추진할지 방향 확인이 필요합니다.

Q2. 표준화의 깊이

소프트웨어 모듈화 시, 표준을 어느 수준까지 맞출 것입니까?

A

국제 표준 수준

외부 협업, Exit(소스코드 매각) 시에도 호환되는 설계 | 투자 비용 높음, 장기적 가치 극대화

B

NS Lab 사내 표준

PureWallet ↔ PureCertificate ↔ PurePet 간 모듈 공유 | 중간 투자, 프로젝트 간 시너지

C

PureWallet 프로젝트 내부

현재 집중 개발 중인 프로젝트 내부만 모듈화 | 최소 투자, 즉시 효과

A. 국제 표준
